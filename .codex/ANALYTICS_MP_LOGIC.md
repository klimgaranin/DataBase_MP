# ANALYTICS_MP_LOGIC

## Источник

Живая Google Таблица `Аналитика МП` из `mp-gas/apps/analytics-mp` является
первым бизнес-эталоном для web-модуля аналитики.

Последний read-only аудит:

`local/audits/sheets/analytics_mp_sheets_audit_20260721-203226.md`

## Что видно по таблице

- `Аналитика WB`: 1659 строк, 164 колонки. В первых 120 строках найдено 7173
  формулы.
- `Аналитика OZON`: 1012 строк, 72 колонки. В первых 120 строках найдено 7891
  формула.
- `DATA`: скрытый лист на 50000 строк, 46 колонок. Это не нормальная БД, а
  импортированный слой данных через `IMPORTRANGE`.
  Пользователь уточнил: `DATA` полностью дублирует внешний файл выгрузки API
  endpoints. Значит при переносе в PostgreSQL этот файл/источник нужно считать
  первичной входной выгрузкой, а не самостоятельной бизнес-формулой таблицы.
- Локальный файл `local/source_exports/Статистика.xlsm` является источником
  для `DATA`/API endpoints. Статический аудит Power Query/VBA сохранён в
  `local/audits/source_files/Статистика_xlsm_powerquery_audit_20260721-205855.md`.
  Внутри 12 Power Query:
  - WB: заказы за 3 месяца, объединённые в один блок; остатки WB;
  - Ozon: заказы, остатки, хранение;
  - Яндекс Маркет: заказы и остатки;
  - внутренние источники: остатки `СМП/ОСН/СОХ/СВХ/ТС`, список заказов в
    производстве/пути/Минске.
- VBA в `Статистика.xlsm` в основном запускает Power Query для заказов WB:
  `Заказы WB1`, `Заказы WB2`, `Заказы WB3`, затем `Заказы WB`, с паузой около
  59 секунд между запросами. Это обход ограничения WB Statistics API, а не
  бизнес-логика. В PostgreSQL/Web это нужно заменить нормальным API-загрузчиком
  с курсором, лимитами и raw-логированием.
- `Остатки ЧЗ`: рабочие блоки без формул:
  - WB `A:E`: артикул, склад, остаток, поставлено с 01.05, остатки не ЧЗ;
  - Ozon `G:K`: артикул, кластер, остаток, поставлено с 01.06, остатки не ЧЗ;
  - WB summary `M:O`: артикул, остаток, товары в пути;
  - Ozon summary `Q:S`: артикул, остаток, товары в пути.
- `ПАНЕЛЬ`: верхние итоги по плану, WB/Ozon/YA, остатку, выручке, расходам РК
  и ДРР.
- `ИНФО`: внутренние производственные/складские статусы по артикулам:
  `СМП`, `ОСН`, `СОХ`, `СВХ`, `ТС`, `План в Минске`, `СОГЛ Заказа`,
  `В ПРОИЗВ`, `ГОТОВ`, `В ПУТИ`.
- `БД`, `Stats_Full_WB`, `Реклама WB`, `Реклама OZON`: рекламные расходы и
  связки кампаний/артикулов.

## Как переносить в PostgreSQL/Web

- `DATA` и внешние `IMPORTRANGE` заменить нормальными источниками БД:
  `raw` -> `staging` -> `core` -> `analytics`.
- `Статистика.xlsm` заменить набором типизированных ingestion-процессов:
  WB/Ozon/Yandex/API/файлы -> raw snapshot -> staging с колонками из отчёта ->
  analytics-витрины.
- `Остатки ЧЗ` переносить как типизированные таблицы:
  `staging.marketplace_stock_locations` и
  `staging.marketplace_stock_summary`.
- `Аналитика WB` и `Аналитика OZON` переносить отдельно. Общие поля можно
  держать в одной таблице `analytics.marketplace_current`, но экран и расчёты
  остаются раздельными по marketplace.
- `ПАНЕЛЬ` строить поверх готовых analytics-витрин, а не поверх сырых API.
- Рекламный блок пока учитывать как источник данных, но не переписывать
  существующий fullstats/GAS-trigger первым этапом.

## Первые витрины

- `analytics.marketplace_current`:
  артикул, название, серия, цена, маржа, конкурентная цена/маржа, остаток,
  остаток не ЧЗ, товары в пути, ADS/ДРР, оборачиваемость и лампочка статуса.
- `analytics.marketplace_product_daily`:
  дневные ряды продаж/заказов/выручки, которые сейчас представлены датированными
  колонками в `Аналитика WB` и `Аналитика OZON`.
- `staging.marketplace_stock_locations`:
  детальный аналог блоков WB/Ozon в `Остатки ЧЗ`.
- `staging.marketplace_stock_summary`:
  агрегированный аналог WB `M:O` и Ozon `Q:S`.
- `staging.advertising_daily`:
  дневные рекламные расходы и заказы для ДРР.
- `core.marketplace_unit_economy`:
  цены и маржинальность по маркетплейсам.
- `core.production_inventory_snapshot`:
  внутренние остатки и производственные статусы из `ИНФО`/source DATA.

## Что показал `Статистика.xlsm`

Для первого рабочего прототипа аналитики нужны входные наборы:

- Заказы WB: `date`, `supplierArticle`, `Кол-во`, `Сумма`, `isCancel`.
  Сейчас Excel делит их на 3 месячных запроса и объединяет в
  `Дата`, `Артикул`, `Кол-во`, `Сумма`, `Отменён`.
- Остатки WB: `Артикул`, `Остаток, шт`; источник в XLSM — старый WB
  Statistics stocks endpoint.
- Заказы Ozon: `Дата`, `Артикул`, `Кол-во`, `Сумма`, `Статус`; источник —
  CSV `orders.csv`.
- Остатки Ozon: `Артикул`, `Остаток, шт`, `В пути, шт`; источник —
  XLSX-отчёт `Остатки OZON.xlsx`.
- Хранение Ozon: `Артикул`, `Платно, шт`, `Платно, л`,
  `Списано в день, RUB`, `Дней до первой платности`.
- Яндекс Маркет: заказы `Дата/Артикул/Кол-во/Сумма` и остатки
  `Артикул/Остаток`.
- Внутренние остатки/производство: `Артикул`, `СМП`, `ОСН`, `СОХ`, `СВХ`,
  `ТС`, `СОГЛ Заказа`, `В ПРОИЗВ`, `ГОТОВ`, `В ПУТИ`, `МИНСК`.

В XLSM обнаружены секреты в Power Query. Они не должны переноситься в код,
отчёты или git; целевое место — `app/secrets.py` с backend `keyring`.
