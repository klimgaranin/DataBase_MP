# DECISIONS

## 2026-07-18 — WB legal export вынесен в архив

Разовая доказательная выгрузка WB не должна оставаться активным production-кодом
проекта. Код, тесты, scripts и evidence перенесены в:

`archive/wb-legal-export-20260718/`

Причина:

- задача была одноразовой;
- активные jobs должны оставаться чистыми;
- evidence нужно сохранить, но не коммитить и не смешивать с ETL.

`archive/` добавлен в `.gitignore`.

## Windows-first для боевых запусков

Проект эксплуатируется на Windows. WSL-копия может использоваться для разработки,
но боевые WB-запросы при проблемах сети нужно запускать через Windows-мост.

Не делать вывод, что API не работает, только по WSL timeout.

## Секреты

`.env` можно читать только когда задача требует диагностики окружения, но нельзя
выводить значения токенов и паролей в ответ.

Новые переменные добавлять в `.env.example`; реальные значения пользователь
заполняет сам.

## Действующие jobs нельзя ломать

Любые изменения вокруг общих модулей `app/db.py`, `app/utils.py`,
`app/clients/*`, `app/normalize/*` должны учитывать действующие jobs:

- `wb_orders`
- `wb_stocks`
- `wb_adv_campaigns`
- `wb_adv_fullstats`
- `api_server`

Узкие одноразовые задачи не должны менять общие utility-функции без необходимости.

## 2026-07-21 — Advertising/fullstats временно не рефакторим

В проекте есть модуль WB advertising fullstats, построенный через API server и
GAS-trigger. Он рабочий, но специфически настроенный и сейчас не срочный.

Решение:

- не переносить его в архив;
- не удалять;
- не включать в первые этапы общего рефакторинга;
- трогать только если это мешает общей работе проекта или пользователь отдельно
  ставит задачу по рекламе/fullstats.

Ближайший рефакторинг вести вокруг общей основы, orders/stocks, зависимостей,
тестов и DB-слоя.

## Миграции

Новые таблицы и изменения схемы должны идти через `migrations/`.
`app/db.py` сейчас содержит часть DDL для `CREATE TABLE IF NOT EXISTS`; это
исторический компромисс, а не желательная новая практика.

## WB API rate limits

Клиенты должны явно учитывать лимиты WB:

- Statistics orders: примерно 1 запрос в минуту.
- Analytics stocks: примерно 3 запроса в минуту.
- Advertising fullstats: батчи по 50 кампаний и пауза около 21 секунды.

Не ускорять боевые выгрузки без проверки документации и последствий.

## 2026-07-21 — Секреты переводим на keyring постепенно

`.env` остаётся для совместимости с действующими Windows jobs. Новый код должен
использовать `app/secrets.py`, где есть backend `env` и `keyring`.

Целевой локальный вариант для Windows-прототипа — Windows Credential Manager
через Python `keyring`. В проверках и будущей админке показывать только статус
секрета: `задан` / `не задан`.

## 2026-07-21 — API-интеграции проектируем как каталог методов

Не создавать отдельную маленькую job на каждый endpoint. Новые интеграции
строить по схеме: API method catalog -> raw JSON в БД -> типизированная staging

Добавлен стартовый каталог `app/integrations/api_catalog.py`, который ссылается
на WB YAML и Ozon swagger из `mp-gas/shared`.

Чтобы не ломать текущие production jobs, новые таблицы аналитики добавляются в

Миграция `V10__marketplace_analytics_foundation.sql` не меняет существующие
таблицы `public.wb_*`; она создаёт фундамент для переноса Google Таблицы:

- raw API ответы и pipeline-запуски;
- общий справочник товаров;
- WB/Ozon товарные соответствия;
- детальные остатки по локациям;
- агрегированные остатки и товары в пути;
- дневные ряды продаж/заказов;
- рекламные дневные агрегаты;
- цены и юнит-экономику;
- внутренние производственные остатки;
- первую web-витрину `analytics.marketplace_current`.

## 2026-07-21 — DATA является дублем внешней выгрузки API endpoints

Пользователь уточнил, что скрытый лист `DATA` полностью задублирован с внешним
файлом выгрузки API endpoints. Поэтому при переносе логики в PostgreSQL:

- `DATA` считать импортированным snapshot-слоем, а не ручной бизнес-таблицей;
- внешний файл/таблицу нужно разобрать как первичный источник;
- для локального `.xlsx`/`.csv` добавлен `tools/source_file_audit.py`;
- рекомендуемое место для файла: `local/source_exports/`, папка игнорируется git.

## 2026-07-21 — `Статистика.xlsm` переносим по форме WB jobs

`Статистика.xlsm` содержит 12 Power Query и VBA-обвязку. VBA в основном
запускает 3 месячных WB orders запроса с паузами и затем объединяет результат;
это обход лимитов, а не бизнес-логика.

Решение:

- WB заказы и остатки не дублировать из Excel, потому что они уже грузятся
  production jobs в `wb_orders_norm` и `wb_stocks_snap`;
- `Статистика.xlsm` использовать как источник недостающих файловых блоков:
  Ozon, Яндекс Маркет, хранение Ozon, внутренние остатки 1С и список заказов;
- оформлять это отдельным job по форме WB jobs: entrypoint, lock, `job_runs`,
  raw snapshot, нормализация, staging tables, лог и TG-алерт;
- секреты, найденные в Power Query, не переносить в код/логи/отчёты.

## 2026-07-21 — Ozon переводим на API, но по слоям БД

Ozon данные из Excel/CSV не должны оставаться главным источником. Целевой
подход:

- товары/SKU получить и хранить один раз в БД;
- другие Ozon API jobs берут нужные SKU/offer_id из БД, а не собирают их
  заново внутри каждого маленького скрипта;
- каждый API-вызов сохраняет raw request/response в `raw.api_responses`;
- полные сущности сохраняются отдельными raw/staging таблицами;
- Google-формат колонок строится потом из SQL/view, а не задаёт структуру raw.

Первый реализуемый шаг — `ozon_orders`: `/v3/posting/fbo/list`. Первый запуск
идёт с 1 января текущего года, дальше job актуализируется каждый час через
`job_cursors` и lookback. Ozon paid placement/storage report
`/v1/report/placement/by-products/create` выгружать отдельным morning job.
Ozon stocks выгружать отдельным job два раза в день.

Яндекс Маркет в текущем этапе закрыт и не трогается. Внутренние остатки 1С и
список заказов остаются файловыми источниками на Windows-путях пользователя.
Для остатков 1С поддерживать только текущий простой формат `.txt` и возможный
старый Excel `.xls`; не расширять форматы без отдельной необходимости.

Power Query M-код из `Статистика.xlsm` использовать как эталон итоговой логики:
какие файлы брать, какие листы/колонки выбирать, как переименовывать, фильтровать
и группировать. Python не обязан повторять M-код построчно, но итоговые данные
должны совпадать с рабочей Excel-логикой.

## 2026-07-23 — Ozon FBO orders актуализируем как текущую сущность с историей

Для Ozon FBO orders рабочий ключ текущего состояния — `posting_number`.
Почасовая job получает период с lookback, сохраняет полный raw payload и
обновляет текущие technical-строки только если payload реально изменился.

История изменений хранится отдельно в `raw.ozon_fbo_posting_versions`: новая
версия создаётся при первом появлении отправления или при изменении полного
JSONB payload. Повторный запуск с тем же ответом API не должен создавать
дубликаты истории.

Для бизнес-проверки статусов и изменений используется view
`analytics.ozon_fbo_posting_change_history_flat`.

## 2026-07-21 — Единый формат jobs

Новые jobs писать в одном стиле с WB jobs:

- отдельный HTTP/file client;
- отдельная нормализация;
- отдельные DB-функции/миграции;
- advisory lock;
- запись в `job_runs`;
- Telegram-алерт по итогам;
- raw request/response logs для API jobs;
- Windows `.cmd` entrypoint для production-запуска.

Если формат нужно улучшать, улучшать сразу как общий стандарт и затем
постепенно переносить существующие jobs, а не делать разнобой.

## 2026-07-21 — Рабочий цикл разработки и внедрения

Для новых production jobs процесс такой:

- разработать и проверить локально в WSL;
- обязательно прогнать тесты и dry-run через Windows-мост;
- после готовности зафиксировать изменения в git;
- затем подтянуть изменения на сервере и проверить серверный запуск;
- если серверная проверка не прошла, исправлять до рабочего состояния, а не
  брать следующую задачу.

## 2026-07-21 — Локальный Windows Docker может иметь старый пароль БД

Если `.env` скопирован с сервера, а локальный контейнер PostgreSQL уже был
создан раньше, `POSTGRES_PASSWORD` из `.env` не меняет пароль внутри старого
Docker volume. Симптом: `docker exec infra-db-1 psql ...` работает, а Python по
`PG_DSN` получает `password authentication failed`.

Решение для локальной разработки:

```powershell
python tools\sync_local_postgres_password.py
```

После этого проверять:

```powershell
python tools\health_check.py --log-lines 0
```

## 2026-07-21 — Ozon stocks повторяет рабочую GAS-логику

В GAS для Ozon stocks рабочая цепочка была:

- `/v3/product/list` по `ALL` и `ARCHIVED`;
- `/v3/product/info/list`;
- `/v1/analytics/stocks` пачками по 100 SKU через batch-запросы;
- retry только для упавших запросов.

Python job должен следовать этой логике, а не делать медленную последовательную
цепочку. Для 429 без `Retry-After` используется длинный backoff, потому что
боевой сервер может держать лимит дольше, чем GAS.

## 2026-07-21 — Raw, detail и витрины не смешиваем

Для API jobs постоянное правило:

- `raw` хранит сырой ответ/страховку и историю запросов;
- `staging.*_details` хранит полные нормализованные строки без бизнес-
  суммирования;
- агрегаты, схлопывания и итоговые суммы должны жить в отдельных summary/
  витринах или SQL-запросах.

`raw.ozon_analytics_stocks` — журнал доказательных API-снимков. Он может
содержать несколько полных снимков за день и поэтому не подходит для простой
суммы остатков без фильтра по `source_run_id`.

`staging.ozon_stock_details` — полные нормализованные строки остатков Ozon.
`staging.ozon_stock_by_cluster` — агрегированная текущая витрина. В ней строки
схлопнуты по `(sku, offer_id, cluster_id)` только для быстрых расчётов.

Ozon FBO statuses обновляются по `posting_number`: raw posting и нормализованные
товарные строки обновляются при повторной выгрузке отправления. Детальные
товарные строки дополнительно имеют `line_number`, чтобы не потерять повторный
SKU внутри одного отправления.

## 2026-07-22 — API payload не является рабочей таблицей

Для всех WB/Ozon endpoint-ов, которые проект выгружает в БД, правило такое:

- сырой API/file ответ хранится отдельно как страховка, аудит и источник для
  повторной нормализации без нового API-запроса;
- рабочие `staging` таблицы и `analytics` views должны раскрывать полезные поля
  в обычные колонки;
- бизнес-расчёты и DBeaver-проверки не должны требовать ручного разбора
  `payload`;
- если API отдаёт вложенный список, рабочий слой должен либо развернуть его в
  отдельные строки, либо явно вынести ключевые поля/счётчики в колонки;
- исторические detail-таблицы нельзя суммировать как текущий остаток без фильтра
  по последнему `source_run_id`/`snapped_at`.

Для Ozon добавлены плоские рабочие слои:

- `analytics.ozon_fbo_order_items_flat` — отправления и товарные строки FBO;
- `analytics.ozon_products_flat` — карточки/товары из product list/info;
- `analytics.ozon_stock_details_flat` — исторические детальные остатки;
- `analytics.ozon_stock_details_latest_flat` — только последний снимок остатков.
- `analytics.ozon_placement_cells_flat` — оригинальные колонки Excel-отчёта
  Ozon placement в виде ячеек, потому что состав колонок отчёта может меняться.

## 2026-07-23 — Ozon FBO orders держим как raw + одна полная техническая таблица

Ozon FBO orders — новый блок, поэтому в нём не оставляем переходные укороченные
таблицы.

Решение:

- `raw.ozon_fbo_postings` хранит полный сырой posting как страховку;
- `staging.ozon_fbo_order_items_full` хранит полную нормализованную техническую
  строку: шапка posting, analytics, legal, external_order, additional_data,
  product, financial product, commission, discounts;
- `analytics.ozon_fbo_order_items_flat` является удобной view поверх полной
  технической таблицы;
- старые `staging.ozon_fbo_posting_details`,
  `staging.ozon_order_item_details`, `staging.ozon_order_items` удалены V20.

Общее правило для новых API-блоков: сначала полный технический слой, потом
обёртки/короткие витрины. Не наоборот.

## 2026-07-22 — Advisory lock должен жить на одном соединении

PostgreSQL session advisory lock принадлежит конкретному соединению. Нельзя
брать `pg_try_advisory_lock` через короткий `with connect()` и затем отпускать
его другим соединением: это даёт Docker warning
`you don't own a lock of type ExclusiveLock` и фактически не защищает job от
параллельного запуска.

Решение: `try_advisory_lock()` в `app/db.py` держит отдельное живое соединение
внутри процесса до вызова `advisory_unlock()`. `advisory_unlock()` отпускает lock
на том же соединении и закрывает его. Это общий механизм для WB/Ozon/source jobs.

## 2026-07-22 — Не использовать сложный `python -c` через PowerShell-мост

PowerShell-кавычки регулярно ломают команды с вложенными строками, JSON,
словарями Python и многострочным кодом. Это не должно быть источником падений
при диагностике.

Правило:

- для сложной проверки создавать временный/служебный `.py` tool и запускать
  `python tools\name.py`;
- для повторяемых проверок добавлять нормальный script/tool в репозиторий;
- `python -c` через `powershell.exe -Command` использовать только для простых
  однострочных команд без вложенных кавычек;
- если всё-таки нужен inline PowerShell, вывод делать простым текстом, а не
  Python dict/JSON literal.

## 2026-07-23 — Локальный AI workflow вместо установки внешнего gstack

Идея gstack полезна проекту как процесс: разные роли для продукта, архитектуры,
ETL, БД, UI, QA и релиза. Но внешний фреймворк не устанавливаем в production-
репозиторий как зависимость.

Решение:

- держать собственный workflow в `.codex/AI_WORKFLOW.md`;
- хранить роли в `.codex/roles/`;
- хранить чеклисты в `.codex/workflows/`;
- проверять наличие workflow через `tools/ai_workflow_check.py`;
- применять роли по задаче: бизнес-фича, ETL/API, БД, админка, release-check.

Причина: проект рабочий, Windows-first и с действующими jobs. Нам нужен
контролируемый процесс без лишних внешних зависимостей и без изменения
production runtime.

## 2026-07-23 — Стандарты структуры и типизации обязательны

Новые файлы, таблицы, jobs, tools и временные артефакты должны создаваться по
стандарту `.codex/PROJECT_STANDARDS.md`.

Решение:

- production-код лежит в `app/`, повторяемые инструменты в `tools/`, тесты в
  `tests/`, миграции в `migrations/`;
- API/file данные проходят слои `raw -> staging -> analytics`;
- raw хранит страховку, staging хранит полный технический слой, analytics
  хранит удобные views;
- `dict[str, Any]` допустим на raw-границе, но новые межмодульные контракты
  должны получать явные типы или покрытую тестом нормализацию;
- временные файлы после задачи удаляются или переносятся в `local/`/`archive/`;
- перед финальным ответом для структурных изменений запускать
  `tools/project_standards_check.py`.

Причина: проект растёт из набора jobs в систему для отдела маркетплейсов, и без
единых правил структура быстро станет непонятной и дорогой в поддержке.
