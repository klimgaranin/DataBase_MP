# Workflow: Change Review

Использовать перед крупным изменением, после багфикса или когда пользователь
просит проверить взрослость/целостность.

## Шаги

1. Посмотреть `git status --short`.
2. Понять, какие файлы уже изменены и не являются ли они чужими изменениями.
3. Прочитать затронутые jobs/clients/normalize/db/migrations/tests.
4. Применить нужные роли:
   - `etl_reviewer` для выгрузок;
   - `db_safety` для БД;
   - `admin_ui_reviewer` для интерфейса;
   - `release_manager` перед финалом.
5. Выписать риски.
6. Исправить реальные проблемы.
7. Прогнать проверки.

## Формат вывода

- Сначала фактические проблемы.
- Затем что исправлено.
- Затем проверки и остаточные риски.
