# AI_WORKFLOW

## Зачем это нужно

Этот проект развивается как рабочая система для отдела маркетплейсов. Пользователь
описывает бизнес-цель простыми словами, а Codex должен доводить её до чистой,
проверенной и безопасной реализации.

Подход взят из идеи `gstack`: не один общий режим "напиши код", а несколько
ролей и проверочных этапов. В проект не устанавливается внешний фреймворк:
правила лежат в `.codex/` и применяются как локальный рабочий процесс.

## Основной цикл

1. Понять бизнес-цель.
2. Прочитать контекст проекта и релевантные файлы.
3. Выбрать нужные роли из `.codex/roles/`.
4. Проверить стандарты размещения и типизации через
   `.codex/PROJECT_STANDARDS.md`.
5. Составить короткий план понятным языком.
6. Реализовать минимально достаточный рабочий вариант.
7. Убрать временное или перенести его в правильное место.
8. Проверить тестами, статикой, БД-smoke или Windows-мостом, если это job.
9. Обновить `.codex/PROJECT_CONTEXT.md`, `.codex/TASKS.md`,
   `.codex/DECISIONS.md`, если изменились архитектура или правила.
10. Дать пользователю простой отчёт: что сделано, где смотреть, как проверить.

## Когда какую роль включать

- Любая новая бизнес-фича: `roles/mp_product_owner.md`,
  `roles/mp_architect.md`.
- Новая или изменённая API/file выгрузка: `roles/etl_reviewer.md`,
  `roles/db_safety.md`.
- Миграции, DB-функции, схемы: `roles/db_safety.md`.
- Админка, фронт, интерфейс менеджера: `roles/admin_ui_reviewer.md`.
- Создание файлов, папок, стандартов, рефакторинг структуры:
  `roles/project_steward.md`, `workflows/cleanup_check.md`.
- Перед пушем, деплоем или боевым запуском: `roles/release_manager.md`.
- Непонятная ошибка в данных: `roles/etl_reviewer.md`,
  `workflows/change_review.md`.

## Жёсткие правила

- Не выводить секреты.
- Не менять `.env` без прямого запроса.
- Не ломать действующие WB jobs и advertising/fullstats.
- Raw хранить как страховку, рабочие таблицы делать раскрытыми в колонки.
- Для production jobs проверять Windows-совместимость через мост.
- Не использовать сложный `python -c` через PowerShell-мост.
- Не начинать боевой API-запуск без теста/dry-run/smoke, если менялась логика.
- Новые файлы, таблицы и tools класть согласно `.codex/PROJECT_STANDARDS.md`.

## Минимальный стандарт результата

Для кода:

- изменения ограничены задачей;
- есть тест или объяснённая причина, почему тест невозможен;
- `python -m compileall app tools tests` проходит;
- `python -m unittest discover -s tests` проходит, если тесты есть;
- `python tools/project_standards_check.py` проходит;
- для jobs есть понятные русские логи и Telegram-итог.

Для данных:

- raw ответ сохранён;
- нормализованная рабочая таблица раскрывает полезные поля;
- есть понятная view для DBeaver/аналитики;
- исторические таблицы не смешаны с текущими витринами;
- контрольные суммы/количества проверены SQL-запросом.
